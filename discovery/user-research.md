# 用户研究报告 — IntelliKnow KMS

## 1. 用户画像

### 画像 A：知识库管理员（Admin）

| 属性 | 描述 |
|------|------|
| **职位** | IT 管理员 / 运营经理 / HR 经理（兼任知识管理） |
| **公司规模** | 50-500 人的中小型企业 |
| **技术水平** | 中等，会用 Web 工具，不会写代码 |
| **使用频率** | 每周 2-3 次（上传文档、维护意图空间、查看报表） |
| **核心目标** | 让员工能快速找到正确信息，减少重复性问题的人工回答 |
| **主要痛点** | 文档散落在邮件/共享盘中；员工总问相同问题；不知道哪些文档被频繁查询 |
| **成功标准** | 上传文档后 5 分钟内可被查询；员工重复性咨询减少 30% |

**典型一天**：
> 早上收到 3 封 HR 政策相关邮件 → 想到如果 KMS 里有这些内容就不用每次手动回答 → 上传新的 HR 政策 PDF → 在意图配置页面确认它被归入 HR 域 → 下午查看分析报表，发现"年假政策"是最高频查询

### 画像 B：普通员工（End User）

| 属性 | 描述 |
|------|------|
| **职位** | 各部门员工（HR、法务、财务、运营） |
| **使用渠道** | Telegram 或 Microsoft Teams（日常工作工具） |
| **技术水平** | 低，只会发消息 |
| **使用频率** | 每天 1-5 次（随需查询） |
| **核心目标** | 快速获得准确答案，不想打扰同事或等待回复 |
| **主要痛点** | 不知道去哪里找信息；找到文档但信息太长难以定位；需要等待人工回复 |
| **成功标准** | 3 秒内收到答案；答案有明确的来源文档引用 |

**典型场景**：
> 员工 A 在 Teams 里问"我还剩几天年假？" → KMS Bot 分类为 HR 意图 → 检索 HR 政策文档 → 回复：根据《员工手册第3章》，年假政策为... [来源: HR-Policy-2024.pdf]

## 2. 核心使用场景

### 场景 1：文档上传与知识库构建（Admin）

```
触发：Admin 获得新的企业文档（政策、流程、规范）
操作：
  1. 登录 Admin Dashboard
  2. 进入 KB Management 页面
  3. 拖拽上传 PDF 或 DOCX 文件
  4. 选择关联的意图空间（HR / Legal / Finance）
  5. 等待系统自动解析（进度指示器）
  6. 确认文档状态变为 "Processed"
期望结果：文档内容可被员工通过 Bot 查询
痛点：希望处理过程透明（知道在解析哪一步）；上传失败时有清晰错误提示
```

### 场景 2：员工通过 Bot 查询（End User）

```
触发：员工在 Telegram / Teams 中有工作相关问题
操作：
  1. 直接在 Bot 对话框发送自然语言问题
  2. 等待回复（期望 ≤ 3 秒）
  3. 阅读带来源引用的回答
期望结果：准确、简洁的答案，附有文档来源
痛点：回答太长或太模糊；没有来源无法验证；Bot 不理解问题时应有明确的兜底提示
```

### 场景 3：意图空间配置与优化（Admin）

```
触发：发现某类查询的分类准确率低 / 需要新增知识域
操作：
  1. 进入 Intent Configuration 页面
  2. 查看查询分类日志（哪些查询被错误分类）
  3. 新建意图空间或编辑描述/关键词
  4. 验证后续查询分类改善
期望结果：可以自定义意图空间并看到效果
```

### 场景 4：Analytics 监控（Admin）

```
触发：周期性检查知识库使用情况
操作：
  1. 进入 Analytics 页面
  2. 查看：最高频查询、最常访问文档、各意图空间查询量分布
  3. 导出报告
期望结果：数据可视化、可导出、帮助决策哪些文档需要更新
```

## 4. 用户研究结论

### 关键洞察

1. **Admin 的核心动机是"减少重复性人工回答"**：上传文档的动力来自于减轻自身负担，而非主动提升知识共享
2. **End User 最在意速度和来源**：回答必须快（≤3秒）且可验证（引用来源），否则不会信任
3. **部署体验是 Admin 的第一个摩擦点**：如果 docker-compose 启动失败，后续功能再好也没有意义 → README 和 .env.example 必须极其清晰
4. **Bot 的"零学习成本"是关键**：End User 不会阅读文档，直接发消息，Bot 必须能处理自然语言

### 设计原则（输入 PRD）

- Admin UI：操作路径不超过3步；文档处理状态实时可见
- Bot 响应：≤3秒；每条回答附文档来源；无匹配时给出明确提示（而非沉默或错误信息）
- 部署：README Quick Start 控制在10行以内；.env.example 每个变量有注释说明
